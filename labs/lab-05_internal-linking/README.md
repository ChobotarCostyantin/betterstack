#### 1.2 - Виявлення orphan pages

**Orphan page** - сторінка на яку не веде жодне внутрішнє посилання. Google може її не знайти навіть якщо вона є в
sitemap.

Перевірити кожну сторінку з інвентарю:

```bash
# Перевірити чи є посилання на конкретну сторінку
# на головній та в категоріях
curl https://твій-домен | grep "/articles/твій-slug"
curl https://твій-домен/categories/javascript | grep "/articles/твій-slug"
```

Заповнити у таблиці колонку "Статус":

```
linked     - є мінімум одне вхідне посилання
orphan     - жодного вхідного посилання
nav-only   - посилання тільки з навігації (не контекстне)
```

#### 1.3 - Аналіз анкорів

Для 5 будь-яких статей свого блогу переглянути всі вихідні посилання та заповнити таблицю (аркуш **"Anchor Audit"**):

| Сторінка-джерело            | Анкор текст               | URL призначення        | Тип анкору  | Оцінка |
|-----------------------------|---------------------------|------------------------|-------------|--------|
| /articles/react-hooks-guide | "тут"                     | /articles/usestate     | generic     | ❌      |
| /articles/react-hooks-guide | "докладніше про useState" | /articles/usestate     | descriptive | ✅      |
| /articles/react-hooks-guide | "JavaScript"              | /categories/javascript | breadcrumb  | ✅      |

Типи анкорів для класифікації:

| Тип               | Опис                    | Приклад                       | SEO оцінка                  |
|-------------------|-------------------------|-------------------------------|-----------------------------|
| **exact-match**   | Точне входження keyword | "react hooks tutorial"        | ⚠️ використовувати обережно |
| **partial-match** | Часткове входження      | "гайд по хукам"               | ✅                           |
| **descriptive**   | Описовий текст          | "як працює useEffect"         | ✅                           |
| **branded**       | Назва сайту або бренду  | "IT Blog"                     | ✅                           |
| **generic**       | Неінформативний         | "тут", "читати", "click here" | ❌                           |
| **naked URL**     | Голий URL як анкор      | "https://…/article"           | ❌                           |
| **breadcrumb**    | Хлібні крихти           | "JavaScript → Hooks"          | ✅                           |

### 2. Побудова схеми перелінковки

#### 2.1 - Принципи схеми для IT блогу

Перед побудовою схеми зафіксувати базові правила для свого проекту:

**Горизонтальна перелінковка (всередині силосу):**

```
Категорія → всі статті цієї категорії
Стаття → 2-4 пов'язані статті тієї самої категорії
Стаття → сторінка категорії (breadcrumb)
```

**Вертикальна перелінковка (між рівнями):**

```
Головна → всі категорії
Головна → featured статті (топ-5 останніх або популярних)
Категорія → головна (footer або breadcrumb)
```

**Перехресна перелінковка (між силосами) - з обережністю:**

```
Стаття про React → стаття про Node.js ✅ (є смисловий зв'язок)
Стаття про React → стаття про кібербезпеку ❌ (немає зв'язку)
```

#### 2.2 - Схема у Google Sheets

На аркуші **"Link Scheme"** побудувати повну схему перелінковки у вигляді таблиці:

| Звідки (URL)                  | Куди (URL)                     | Анкор текст                  | Тип посилання | Розміщення на сторінці  | Пріоритет |
|-------------------------------|--------------------------------|------------------------------|---------------|-------------------------|-----------|
| `/`                           | `/categories/javascript`       | "JavaScript"                 | nav           | header navigation       | high      |
| `/`                           | `/articles/react-hooks-guide`  | "React Hooks - повний гайд"  | contextual    | featured articles block | high      |
| `/categories/javascript`      | `/articles/react-hooks-guide`  | "Повний гайд по React Hooks" | contextual    | article listing         | medium    |
| `/articles/react-hooks-guide` | `/categories/javascript`       | "JavaScript"                 | breadcrumb    | breadcrumb nav          | high      |
| `/articles/react-hooks-guide` | `/articles/usestate-guide`     | "детальніше про useState"    | contextual    | article body            | medium    |
| `/articles/react-hooks-guide` | `/articles/useeffect-patterns` | "патерни useEffect"          | related       | related articles block  | low       |

Типи розміщення:

- `header navigation` - головне меню
- `breadcrumb nav` - хлібні крихти
- `article listing` - список статей у категорії
- `article body` - всередині тексту статті (найцінніші)
- `related articles` - блок пов'язаних статей
- `footer` - підвал сайту
- `sidebar` - бічна панель

**Мінімальна вимога:** не менше **20 посилань** у схемі.

## Контрольні питання

### Рівень 1 - Розуміння термінів

1.  **PageRank та передача "ваги"**:
    **PageRank** — це алгоритм Google, який оцінює авторитетність сторінки на основі кількості та якості посилань, що на неї ведуть. Внутрішня перелінковка працює як "система каналів": вона розподіляє "вагу" (link juice) від найбільш авторитетних сторінок (зазвичай Головної) до глибших розділів. Чим більше посилань веде на сторінку, тим важливішою вона здається пошуковій системі.
2.  **Orphan page та Sitemap**:
    **Orphan page** — це сторінка, на яку не веде жодне внутрішнє посилання. Навіть якщо вона є в Sitemap, Google може сприймати її як ізольовану або неактуальну. Пошукові роботи використовують посилання для розуміння структури та контексту; без них сторінка позбавлена "зв'язку" з основною тематикою сайту.
3.  **`rel="nofollow"` vs `rel="noopener"`**:
    * `rel="nofollow"`: Каже пошуковому роботу не передавати "вагу" (PageRank) сторінці, на яку веде посилання. Використовується для платних лінків або підозрілого контенту.
    * `rel="noopener"`: Це атрибут безпеки для посилань з `target="_blank"`. Він не дає новій вкладці отримати доступ до вікна, з якого було зроблено перехід. На SEO він не впливає, але критичний для безпеки.
4.  **Цінність Contextual links**:
    Контекстні посилання (всередині тексту) цінніші, тому що вони оточені релевантним текстом (навколопосиланковим текстом). Google бачить, що посилання логічно доповнює статтю, а не просто висить у меню для навігації. Це сигнал про високу релевантність.
5.  **Crawl depth**:
    Це кількість кліків, необхідна для переходу з головної сторінки до цільової. Максимально прийнятним значенням вважається **3 кліки**. Все, що глибше, індексується гірше та отримує дуже мало "ваги".

### Рівень 2 - Аналіз

6.  **50 посилань на сторінці категорії**:
    Кожна сторінка має обмежений обсяг "ваги", яку вона може передати. Ця вага ділиться порівну між усіма посиланнями. Якщо їх 50, то кожне отримує лише **1/50** частку. Це не є прямою проблемою (якщо це зручно юзеру), але ефективність передачі авторитету кожній статті значно падає.
7.  **Порівняння анкорів**:
    Варіант **(б) "як працюють замикання в JavaScript"** кращий. Чому?
    * Він містить ключові слова, за якими сторінка-призначення хоче ранжуватися.
    * Він дає боту чітке розуміння контексту сторінки, на яку веде лінк. Анкор "читати тут" — порожній для алгоритму.
8.  **Відсутність лінків між статтями в Silo**:
    Це розриває тематичний кластер. Замість того, щоб створити "хмару авторитету" навколо теми JavaScript, ці статті існують окремо. Google гірше розуміє взаємозв'язок між ними, і категорія в цілому отримує менше тематичного трасту.
9.  **Посилання на себе**:
    Це називається "циклічним посиланням". Це не є критичною помилкою або "пенальті", але це **марна трата PageRank**. Ви віддаєте вагу сторінці, яка її вже має, замість того, щоб "підсвітити" інші важливі розділи.
10. **Header vs Contextual лінк**:
    **Contextual** посилання в тілі статті вважається сильнішим для передачі релевантності конкретній темі. **Header navigation** лінки важливі для передачі загального авторитету домену на головні хаби (категорії), але вони мають меншу вагу в плані розкриття конкретного "інформаційного інтенту".

### Рівень 3 - Синтез та висновки

11. **Аналіз власної схеми**:
    Найбільше посилань зазвичай отримують **Категорії** (через меню та breadcrumbs) та **Головна**. Для SEO-стратегії це правильно, адже ці сторінки акумулюють вагу і розподіляють її далі. Проте, якщо стратегія — просувати конкретну "жирну" статтю, її треба виносити в блок "Featured" на головну.
12. **Розподіл PageRank (умовний приклад)**:
    * Головна: 10 PR. Посилається на 5 категорій -> кожна категорія отримує по **2 PR**.
    * Категорія: 2 PR. Посилається на 10 статей -> кожна стаття отримує по **0.2 PR**.
    * Найменше отримає стаття, що знаходиться на 4+ рівні вкладеності без контекстних лінків із "сильних" сторінок.
13. **Аналіз DOU/AIN**:
    Варто запозичити систему **внутрішніх тегів**. На DOU під кожною статтею є теги (напр. "Кар'єра", "Frontend"). Це створює величезну кількість додаткових перехресних лінків між силосами, що дозволяє боту легко "гуляти" по сайту і збирати весь контент.
14. **Серії статей (Courses)**:
    Схема зміниться на **лінійну + ієрархічну**:
    * Кожна стаття серії повинна мати кнопки "Попередня" та "Наступна" (лінійна перелінковка).
    * Усі вони мають посилатися на "Головну сторінку курсу" (Hub).
    * Це створює надпотужний тематичний вузол, де PageRank циркулює по колу від початку до кінця курсу.
